좋은 방향입니다. **기업용 API 상품화를 시작할 수 있는 뼈대는 잡혀 있습니다.**
다만 현재 설계는 “외부에서 호출 가능한 API” 수준이고, **기업 고객에게 제공 가능한 API 제품**으로 보려면 보안·권한·과금·격리·감사·운영 안정성 쪽 보완이 필요합니다.

핵심 보완점은 아래입니다.

---

# 1. 가장 중요한 보완: API Key만으로는 기업용 보안이 약함

현재 설계는 `Authorization: Bearer <api_key>` 방식입니다. MVP로는 괜찮습니다. 하지만 기업 고객 대상이면 API Key 하나가 유출됐을 때 피해 범위가 너무 큽니다.

## 보완 필요

API Key 테이블에 아래 필드를 추가하세요.

```ts
agent_api_keys
- id
- name
- key_hash
- key_prefix
- owner_user_id
- organization_id
- scopes
- allowed_origins
- allowed_ips
- expires_at
- last_used_at
- revoked_at
- is_active
- created_at
```

특히 중요합니다.

```txt
organization_id
scopes
allowed_ips
key_prefix
last_used_at
revoked_at
```

## 이유

기업 고객은 보통 이렇게 요구합니다.

```txt
이 키는 A 고객사 전용인가?
이 키로 어떤 시나리오만 호출 가능한가?
IP 제한이 가능한가?
누가 언제 호출했는가?
키 유출 시 즉시 폐기 가능한가?
```

현재 설계에는 이 부분이 부족합니다.

---

# 2. API Key 원문 저장 금지는 맞지만, `key_prefix`가 필요함

`key_hash`만 저장하는 건 맞습니다. 하지만 관리 화면에서 키를 구분할 수 있어야 합니다.

예:

```txt
rpb_live_8f3a****************
```

## 보완안

API 키 생성 규칙:

```txt
rpb_live_<public_prefix>_<secret>
rpb_test_<public_prefix>_<secret>
```

DB에는:

```ts
key_hash: sha256(fullKey)
key_prefix: "rpb_live_8f3a"
```

관리 화면에는 `key_prefix`만 보여줍니다.

---

# 3. 권한 범위, 즉 scopes가 필요함

현재 API 키는 발급되면 모든 agent API를 호출할 수 있는 구조입니다. 기업용으로는 위험합니다.

## 권장 scopes

```txt
scenarios:read
personas:read
sessions:create
sessions:read
sessions:message
sessions:end
usage:read
```

예:

```json
{
  "scopes": [
    "scenarios:read",
    "sessions:create",
    "sessions:message"
  ]
}
```

나중에 고객사별로 이렇게 제한할 수 있습니다.

```txt
A 고객사 키: 특정 온보딩 시나리오만 사용 가능
B 고객사 키: 페르소나 자유 대화만 사용 가능
C 고객사 키: 조회만 가능
```

---

# 4. 고객사/조직 단위 멀티테넌시가 빠져 있음

현재 `owner_user_id` 기준으로 되어 있습니다. 하지만 기업 API는 사용자 개인보다 **조직 단위**가 더 중요합니다.

## 보완안

```ts
agent_api_keys.organization_id
agent_sessions.organization_id
agent_usage.organization_id
```

또는 기존 시스템에 organization 개념이 있다면 반드시 연결하세요.

## 이유

기업 고객별로 다음을 분리해야 합니다.

```txt
API 키
사용량
시나리오 접근 권한
세션 로그
과금
관리자
데이터 보존 정책
```

`owner_user_id`만 있으면 나중에 B2B 과금/권한 관리가 불편해집니다.

---

# 5. 세션 모델을 기존 conversations에 억지로 넣기보다 `agent_sessions`를 고려해야 함

현재 계획은 기존 conversations 테이블에 `source: 'agent_api'`를 추가하는 방식입니다. 빠르게 만들기는 좋습니다.

하지만 외부 API 세션은 브라우저 사용자 대화와 성격이 다릅니다.

## 선택지

### MVP

기존 `persona_runs` 또는 conversations 구조 재사용 + `source='agent_api'`

괜찮습니다.

### 기업용 안정화

별도 테이블 권장:

```ts
agent_sessions
- id
- agent_key_id
- organization_id
- external_user_id
- persona_run_id
- scenario_run_id
- scenario_id
- persona_id
- language
- difficulty
- status
- metadata
- created_at
- ended_at
```

## 이유

외부 고객사의 `userId`와 내부 서비스의 `userId`를 섞으면 안 됩니다.
외부 API의 `userId`는 내부 계정 ID가 아니라 **externalUserId**로 처리해야 합니다.

현재 `POST /sessions`의 `userId`는 이름을 바꾸는 게 좋습니다.

```txt
userId → externalUserId
```

# 6. 외부 userId를 신뢰하면 안 됨

현재 세션 시작 파라미터가:

```txt
scenarioId, personaId, userId, difficulty, language
```

인데, 외부에서 `userId`를 넘기면 내부 사용자 ID처럼 오해될 수 있습니다.

## 보완안

```json
{
  "scenarioId": "customer-complaint-001",
  "personaId": "angry-customer",
  "externalUserId": "slack-user-U12345",
  "externalSessionId": "optional-client-session-id",
  "difficulty": 4,
  "language": "ko",
  "metadata": {
    "channel": "slack",
    "teamId": "T123",
    "department": "sales"
  }
}
```

내부 DB에는 이렇게 저장하세요.

```txt
external_user_id
external_session_id
metadata jsonb
```

# 7. 시나리오 접근 권한 제한이 필요함

현재:

```txt
GET /api/v1/agent/scenarios
```

가 모든 활성 시나리오를 반환하는 것으로 보입니다.

기업용 API에서는 고객사별로 노출 가능한 시나리오가 달라야 합니다.

## 보완안

API 키 또는 organization에 허용 시나리오를 연결하세요.

```ts
agent_api_key_scenarios
- agent_key_id
- scenario_id
```

또는:

```ts
organization_scenarios
- organization_id
- scenario_id
```

응답은 반드시 권한이 있는 것만 반환해야 합니다.

```txt
공개 시나리오 전체 노출 금지
```

# 9. Idempotency-Key가 필요함

외부 시스템은 네트워크 오류 때문에 같은 요청을 재시도합니다.
`POST /sessions`나 `POST /messages`가 중복 실행되면 세션이나 메시지가 두 번 생성될 수 있습니다.

## 보완안

헤더 지원:

```txt
Idempotency-Key: <unique-client-generated-key>
```

적용 대상:

```txt
POST /api/v1/agent/sessions
POST /api/v1/agent/sessions/:id/messages
POST /api/v1/agent/sessions/:id/end
```

DB 테이블:

```ts
agent_idempotency_keys
- key
- agent_key_id
- request_hash
- response_body
- status_code
- created_at
```

MVP에서 복잡하면 최소한 `messages`에만 먼저 적용하세요.

---

# 10. Rate limit은 단순 60 RPM만으로 부족함

분당 60 요청은 시작점으로 괜찮습니다. 하지만 기업용은 아래 3개 제한이 필요합니다.

```txt
요청 수 제한
동시 세션 수 제한
월 사용량 제한
```

## 보완안

API Key 또는 organization 단위로:

```ts
rate_limit_per_minute
daily_request_limit
monthly_token_limit
max_concurrent_sessions
```

응답 헤더:

```txt
X-RateLimit-Limit
X-RateLimit-Remaining
X-RateLimit-Reset
Retry-After
```

429 응답 예:

```json
{
  "error": {
    "code": "rate_limit_exceeded",
    "message": "API rate limit exceeded.",
    "retryAfterSec": 30
  },
  "requestId": "req_..."
}
```

하지만 기업용 API 상품화를 생각하면 별도 usage summary가 필요합니다.

```ts
- agent_key_id
- date
- request_count
- session_count
- input_tokens
- output_tokens
- total_tokens

```

실시간 로그는 `ai_usage_logs`, 대시보드는 `agent_usage_daily`가 담당하는 구조가 좋습니다.

# 13. 세션 종료/만료 정책이 필요함

외부 에이전트가 `POST /sessions/:id/end`를 호출하지 않을 수 있습니다.

## 보완안

세션 자동 만료 정책:

```txt
마지막 메시지 후 30분 비활성 → expired
생성 후 24시간 경과 → expired
ended/expired 세션에는 messages 불가
```

DB 필드:

```ts
last_activity_at
expires_at
status: active | ended | expired | failed
```

---

# 14. 입력 검증과 콘텐츠 제한이 필요함

기업 API는 외부 입력이 바로 들어옵니다.

## 보완안

메시지 제한:

```txt
message 최대 길이: 예 4,000자
metadata 최대 크기: 예 8KB
한 세션 최대 턴 수: 예 30턴 또는 시나리오 설정값
language enum 제한: ko/en/ja/zh
difficulty 범위: 1–5
```

그리고 금지:

```txt
HTML/script 그대로 저장 또는 표시 금지
metadata에 민감정보 넣지 말라는 문서 고지
```

# 15. CORS보다 서버-서버 API 중심으로 설계해야 함

외부 API 키는 브라우저에 노출되면 안 됩니다.
따라서 API 문서에 명확히 써야 합니다.

```txt
Agent API keys are intended for server-to-server use only.
Do not expose API keys in browser or mobile client code.
```

모바일 앱에서 직접 호출해야 한다면 별도 short-lived token 교환 방식이 필요합니다.

# 최종 판단

**이 계획은 방향이 좋습니다.**
하지만 지금 상태는 “API를 열 수 있다” 수준이고, 기업용으로 제공하려면 아래 10개는 반드시 보완하세요.

```txt
1. organization_id 기반 멀티테넌시
2. API key scopes
3. key_prefix + last_used_at + revoked_at
4. allowed_ips 또는 allowed_origins
5. externalUserId / externalSessionId 분리
6. 시나리오 접근 권한 제한
7. 표준 에러 포맷 + requestId
8. Idempotency-Key
9. 세션 만료 정책
10. agent_usage_daily 또는 usage summary
```

한 줄로 정리하면:

> **MVP는 가능하지만, 기업용 API 상품으로 팔려면 “보안·권한·격리·사용량·감사 로그”를 반드시 설계에 넣어야 합니다.**
