결론부터 말하면, **이 첨부안으로 개발을 시작해도 됩니다.**
이전 버전에서 문제였던 대부분의 리스크가 반영되어 있습니다. 특히 아래 6가지는 개발 지시서로 충분히 좋습니다.

* 서버 Simulation Engine을 상태의 단일 소유자로 둔 점
* `applySimulationPatch()` 단일 경로로 상태 변경을 제한한 점
* Tool args를 절대값이 아니라 delta/intent 방식으로 바꾼 점
* `persona_runs.simulation_state` + `simulation_events` 이중 저장 구조를 잡은 점
* Tool Call이 없을 때도 서버 rule fallback으로 상태 패널이 움직이게 한 점
* 테스트 항목까지 포함한 점 

다만 **바로 개발 착수 전 보완해야 할 부분이 아직 있습니다.** 아래 항목은 실제 구현 중 문제가 될 가능성이 높습니다.

# 2. `conversationId`와 `personaRunId` 명칭 혼동이 아직 남아 있음

계획에는 `conversationId`, `personaRunId`가 같이 나옵니다.

예:

```ts
interface SimulationUpdateMessage {
  conversationId: string;
  personaRunId: string;
}
```

현재 구조상 실제 대화 단위 ID는 거의 `personaRunId`처럼 쓰입니다. 그런데 프론트에서는 여전히 `conversationId`라고 부를 가능성이 큽니다.

이걸 정리하지 않으면 WebSocket, API, DB 저장에서 ID가 섞입니다.

## 보완안

명확히 적으세요.

```txt
MVP에서 conversationId는 personaRunId와 동일한 값으로 취급한다.
서버 내부 canonical id는 personaRunId이다.
프론트 prop 이름은 기존 호환을 위해 conversationId를 유지할 수 있으나, API payload에는 personaRunId를 항상 포함한다.
```

또는 더 강하게:

```txt
Simulation Engine 내부에서는 conversationId라는 이름을 사용하지 않는다.
항상 personaRunId를 사용한다.
```

추천은 후자입니다.

# 3. `SimulationStatePatch` 타입이 delta 의미와 충돌함

현재 타입은 이렇게 되어 있습니다.

```ts
interface SimulationStatePatch {
  npcEmotions?: Partial<NpcEmotions>; // delta 적용 후 clamp
  stage?: ScenarioStage;
  pressureLevel?: number;
}
```

그런데 `npcEmotions`라는 이름은 절대값처럼 보입니다. 실제로는 delta입니다.

## 보완안

타입 이름을 명확히 바꾸세요.

```ts
interface SimulationStatePatch {
  npcEmotionDelta?: Partial<Record<keyof NpcEmotions, number>>;
  targetStage?: ScenarioStage;
  pressureDelta?: -1 | 0 | 1;
  timerDelta?: Partial<TimerDelta>;
  incidentsToAdd?: Incident[];
  turnScoresToAdd?: TurnScore[];
}
```

이렇게 해야 개발자가 실수로 아래처럼 절대값을 넣지 않습니다.

```ts
npcEmotions: { anger: 90 }
```

현재 계획은 “delta 방식”을 강조하고 있지만, 타입 이름이 아직 절대값처럼 보입니다.

# 4. `turnScores 평균이 가중치대로 계산` 테스트 문구가 모순됨

계획에는 currentScore를 이렇게 정의했습니다.

```txt
MVP: 평가 완료 TurnScore.total의 단순 평균
```

그런데 테스트 항목에는 이렇게 되어 있습니다.

```txt
turnScores 평균이 가중치대로 계산되는지
```

여기서 가중치는 `TurnScore.total` 내부 계산에 적용되는 가중치입니다.
반면 여러 턴의 `currentScore`는 단순 평균입니다.

## 보완안

테스트 문구를 분리하세요.

```txt
- TurnScore.total이 clarity 25%, empathy 20%, logic 20%, ownership 20%, actionPlan 15% 가중치로 계산되는지
- currentScore가 평가 완료 TurnScore.total의 단순 평균으로 계산되는지
- 평가 생략 턴은 currentScore 평균에서 제외되는지
```

이렇게 해야 구현자가 헷갈리지 않습니다.

# 5. `fast mode`에서 API 응답 형태가 불명확함

현재 fast mode는 이렇게 되어 있습니다.

```txt
AI 응답과 evaluateUserResponse() 병렬 실행 → AI 응답 먼저 표시 → 평가 완료 시 simulation_update로 패널 갱신
```

텍스트/TTS는 WebSocket이 항상 열려 있지 않을 수 있습니다. 그러면 평가 완료 후 `simulation_update`를 어디로 보낼지 애매합니다.

## 보완안

텍스트/TTS fast mode의 업데이트 채널을 명확히 해야 합니다.

선택지는 3개입니다.

### 선택 A — 텍스트 모드도 WebSocket 유지

```txt
텍스트/TTS 대화 화면에서도 simulation_update 수신용 WebSocket을 유지한다.
```

### 선택 B — polling

```txt
평가 진행 중이면 클라이언트가 /api/conversations/:id/simulation-state를 짧게 polling한다.
```

### 선택 C — fast mode에서는 응답에 즉시 평가를 포함하지 않음

```txt
AI 응답에는 기존 state를 반환하고, 다음 메시지 요청/GET 재조회 시 최신 state를 받는다.
```

추천은 **A**입니다.
음성뿐 아니라 텍스트/TTS에서도 SimulationPanel이 실시간으로 움직이려면 공통 WebSocket 채널이 있는 게 좋습니다.

추가 문구:

```txt
텍스트/TTS 모드에서도 simulation_update 수신을 위해 lightweight WebSocket 또는 SSE 채널을 사용한다.
MVP에서 구현 부담이 크면 fast mode는 API 응답 동기 평가로 제한한다.
```

# 8. `serverContent.turnComplete`가 사용자 턴과 AI 턴 모두에 걸릴 가능성 확인 필요

계획은 `turnComplete`에서 userTranscriptBuffer 확정 후 평가한다고 되어 있습니다. 실무적으로는 이 이벤트가 모델 응답 완료와도 연관될 수 있습니다.

따라서 조건을 더 엄격히 해야 합니다.

## 보완안

```txt
음성 평가 트리거 조건:
- serverContent.turnComplete 수신
- session.userTranscriptBuffer.trim().length > 0
- 직전 확정 turnId와 동일하지 않음
- 현재 AI output 처리 중이 아님 또는 사용자 발화 확정 플래그가 true
```

중복 평가 방지 필드도 추가하세요.

```ts
session.lastEvaluatedUserTurnId: string | null
```

이걸 안 넣으면 같은 사용자 발화가 두 번 평가될 수 있습니다.

# 9. `recentTurnScores`만 저장하면 리포트에서 전체 점수 접근 방식이 필요함

계획은 `SimulationState`를 snapshot 중심으로 하고, 전체 이력은 `simulation_events`에서 읽게 되어 있습니다. 맞습니다.

그런데 `TurnScore`가 `simulation_events.result` 안에만 들어가면 조회가 불편할 수 있습니다.

## 보완안

둘 중 하나를 선택하세요.

### 선택 A — simulation_events만 사용

```txt
auto_evaluation 이벤트의 result.turnScore에 TurnScore 전체를 저장한다.
리포트 생성 시 eventType='auto_evaluation'만 조회한다.
```

### 선택 B — 별도 turn_scores 테이블 생성

```txt
turn_scores 테이블 신설
personaRunId, turnId, scores, hint, method, confidence
```

MVP는 A가 낫습니다.
그 대신 문서에 명확히 추가하세요.

```txt
TurnScore의 전체 이력은 simulation_events 중 eventType='auto_evaluation'의 result.turnScore에서 복원한다.
```

# 10. 감정 변화 타임라인을 만들려면 stateBefore/After 기준을 명확히 해야 함

리포트에서 감정 변화 타임라인을 만들려면 이벤트별 `stateBefore`, `stateAfter`가 필요합니다. 계획에 포함되어 있습니다.

다만 여러 이벤트가 한 턴에 생기면 어떤 값을 턴 대표값으로 쓸지 정해야 합니다.

## 보완안

```txt
감정 타임라인은 각 turnIndex의 마지막 includeInReport=true 이벤트의 stateAfter.npcEmotions를 대표값으로 사용한다.
해당 턴에 이벤트가 없으면 이전 턴 값을 carry-forward한다.
```

이걸 안 정하면 라인 차트가 들쑥날쑥하거나 중복 포인트가 생깁니다.

# 12. `server_rule` fallback과 `trigger_incident` 조건 중복 정리 필요

계획에는 서버 rule fallback이 있고, trigger_incident 기준도 있습니다.

예를 들어:

```txt
3턴 연속 actionPlan<40
```

이건 Gemini가 trigger_incident를 요청할 수도 있고, 서버 rule이 직접 발생시킬 수도 있습니다.

## 보완안

우선순위를 정하세요.

```txt
Incident 발생 우선순위:
1. Gemini trigger_incident 요청이 유효하면 서버 승인 후 발생
2. Gemini 요청이 없고 서버 조건이 충족되면 server_rule로 incident 발생
3. 같은 turnId에서 이미 incident가 발생했으면 추가 발생 금지
```

그리고 `triggeredBy`로 구분합니다.

```txt
gemini_tool | server_rule
```

오래된 version의 incident는 다음 조건을 모두 만족할 때만 표시:
