# 총평

## 좋은 점

이번 버전에서 가장 좋아진 부분은 6가지입니다.

1. **서버 Simulation Engine을 상태의 canonical owner로 둔 점**
2. **Gemini와 클라이언트는 patch request만 보내도록 제한한 점**
3. **Tool args를 delta/intent 기반으로 설계한 점**
4. **incident message를 서버 템플릿으로 생성하게 한 점**
5. **Tool Call이 없어도 서버 rule fallback으로 상태가 움직이게 한 점**
6. **snapshot 중심 SimulationState + simulation_events 이력 저장 구조를 잡은 점**

이 정도면 “대화형 챗봇”이 아니라 **교육용 NPC 시뮬레이션 엔진**으로 확장하는 방향이 명확합니다.

## 2. `conversationId`와 `personaRunId` 명칭을 정리해야 합니다

첨부안에는 여전히 `conversationId`와 `personaRunId`가 같이 나옵니다. 예를 들어 `SimulationUpdateMessage`에 둘 다 들어갑니다. 

현재 RoleplayBot 구조에서는 실질적인 대화 단위가 `personaRunId`에 가깝습니다. 그래서 내부에서는 하나로 통일하는 게 좋습니다.

## 권장 문구

```txt
Simulation Engine 내부에서는 conversationId를 사용하지 않는다.
모든 상태 저장, 이벤트 저장, 평가, 리포트 연결의 기준 ID는 personaRunId다.

프론트의 기존 prop 이름이 conversationId라면 호환을 위해 유지할 수 있지만,
API payload와 DB, Simulation Engine에서는 personaRunId를 canonical id로 사용한다.
```

이걸 안 정리하면 나중에 WebSocket, API, DB 이벤트가 서로 다른 ID를 바라볼 수 있습니다.

## 3. `SimulationStatePatch` 타입 이름이 아직 절대값처럼 보입니다

현재 첨부안에는 이렇게 되어 있습니다.

```ts
interface SimulationStatePatch {
  npcEmotions?: Partial<NpcEmotions>; // delta 적용 후 clamp
  stage?: ScenarioStage;
  pressureLevel?: number;
}
```

그런데 `npcEmotions`, `pressureLevel`은 이름만 보면 절대값처럼 보입니다.
하지만 설계 원칙은 delta 방식입니다.

## 더 안전한 타입

```ts
interface SimulationStatePatch {
  npcEmotionDelta?: {
    angerDelta?: number;
    trustDelta?: number;
    confusionDelta?: number;
    interestDelta?: number;
  };
  targetStage?: ScenarioStage;
  pressureDelta?: -1 | 0 | 1;
  timerDelta?: Partial<TimerState>;
  incidentsToAdd?: Incident[];
  turnScoresToAdd?: TurnScore[];
}
```

이렇게 바꾸는 게 좋습니다.
개발자가 실수로 `anger: 90` 같은 절대값 업데이트를 넣는 일을 줄일 수 있습니다.

## 4. 텍스트/TTS fast mode의 업데이트 채널이 불명확합니다

첨부안은 fast mode에서 이렇게 되어 있습니다.

```txt
AI 응답과 evaluateUserResponse() 병렬 실행
→ AI 응답 먼저 표시
→ 평가 완료 시 simulation_update로 패널 갱신
```

문제는 **텍스트/TTS 모드에서 WebSocket이 항상 열려 있는지**입니다. 음성 모드는 WebSocket이 있지만, 텍스트/TTS는 일반 POST 응답만 쓰고 있을 수 있습니다.

## 선택해야 합니다

### 추천 A: 텍스트/TTS에서도 lightweight WS 또는 SSE 사용

```txt
텍스트/TTS 대화 화면에서도 simulation_update 수신용 WebSocket 또는 SSE 채널을 유지한다.
```

가장 좋습니다. 실시간 패널 컨셉과 맞습니다.

### 대안 B: MVP에서는 fast mode 비활성화

```txt
MVP에서는 텍스트/TTS 평가를 동기 처리한다.
POST /messages 응답에 AI 응답과 simulationState를 함께 반환한다.
fast mode는 Phase 2.5로 미룬다.
```

개발 난이도를 낮추려면 B가 낫습니다.

# 추가로 다듬으면 좋은 부분

## 6. incident 쿨다운을 전체/타입별로 나누세요

현재는 60초 쿨다운만 있습니다.

더 안전한 규칙:

```txt
전체 incident 쿨다운: 60초
같은 IncidentType 재발생 쿨다운: 120초
같은 turnId에서 incident는 최대 1개
```

예를 들어 `executive_join`이 한 번 발생했는데 70초 뒤 또 임원이 들어오는 것은 어색합니다. 같은 타입은 더 길게 막는 게 좋습니다.

---

## 7. 음성 평가 중복 방지 필드가 필요합니다

`serverContent.turnComplete` 기준으로 평가하면 같은 transcript가 중복 평가될 가능성이 있습니다.

세션에 아래 필드를 추가하는 게 좋습니다.

```ts
lastEvaluatedUserTurnId: string | null;
lastFinalizedUserTranscriptHash: string | null;
```

평가 조건:

```txt
userTranscriptBuffer.trim()이 있고
turnId가 lastEvaluatedUserTurnId와 다르고
동일 transcript hash가 아니면 평가 실행
```

---

## 8. TurnScore 저장 위치를 명확히 하세요

SimulationState는 최근 10개만 저장하고, 전체 이력은 `simulation_events`에 저장합니다. 좋습니다.

그럼 문서에 이 문장을 추가하세요.

```txt
TurnScore 전체 이력은 simulation_events 중 eventType='auto_evaluation'의 result.turnScore에서 복원한다.
SimulationState.recentTurnScores는 UI 표시용 최근 10개 캐시일 뿐이다.
```

그래야 리포트 구현자가 어디서 전체 점수를 읽을지 헷갈리지 않습니다.

---

## 9. 감정 타임라인 대표값 기준이 필요합니다

한 턴에 이벤트가 여러 개 생기면 감정값이 여러 번 바뀝니다.

리포트 기준을 정하세요.

```txt
감정 타임라인은 각 turnIndex의 마지막 includeInReport=true 이벤트의 stateAfter.npcEmotions를 대표값으로 사용한다.
해당 turnIndex에 이벤트가 없으면 직전 값을 carry-forward한다.
```

---

## 10. `admin` 타입은 예약만 한다면 입력에서는 막아야 합니다

타입에는 `admin`이 있습니다. 하지만 MVP에서 admin UI/API는 미구현입니다.

따라서 Zod schema에서는 외부 입력으로 `admin`을 받으면 안 됩니다.

```txt
MVP에서 triggeredBy='admin'은 서버 내부 예약값이다.
외부 Tool/API 입력에서는 gemini_tool, server_rule만 허용한다.
```
