첨부안은 이전보다 훨씬 좋아졌습니다. 특히 **“Gemini는 트리거, 서버 Simulation Engine이 상태 계산·검증·저장·평가 담당”**이라는 원칙이 들어간 점은 핵심을 제대로 잡았습니다. 다만 아직 개발 지시서로 쓰기에는 몇 가지 빠진 부분이 있습니다. 아래를 보완하면 실제 구현 실패 확률이 크게 줄어듭니다.

# 1. 가장 먼저 고쳐야 할 부분: `conversations 테이블` 기준이 애매함

첨부안에는 `simulationState는 메모리 + DB(conversations 테이블 jsonb 컬럼)`에 저장한다고 되어 있습니다. 그런데 현재 RoleplayBot 구조는 최근 코드 기준으로 **personaRuns / scenarioRuns 중심**입니다. 대화 ID도 실제로는 `personaRunId`처럼 쓰이고 있습니다.

따라서 `conversations 테이블`이라고 쓰면 개발자가 잘못된 테이블에 붙일 수 있습니다.

## 보완안

이렇게 명확히 바꾸는 게 좋습니다.

```txt
simulationState는 persona_runs.simulation_state jsonb 컬럼에 최신 스냅샷으로 저장한다.
simulation_events는 persona_run_id를 FK로 저장한다.
scenario 전체 단위 집계가 필요하면 scenario_run_id도 함께 저장한다.
```

추천 스키마:

```ts
personaRuns.simulationState jsonb
```

```ts
simulationEvents {
  id
  personaRunId
  scenarioRunId
  turnIndex
  eventType
  toolName
  args
  result
  stateBefore
  stateAfter
  createdAt
}
```

이유는 간단합니다.
**실제 실시간 음성 세션, 텍스트 메시지, 피드백 리포트가 personaRun 단위로 움직이기 때문**입니다.

# 2. `simulation_events`에 `eventType`이 빠져 있음

첨부안의 `simulation_events`는 `tool_name` 중심입니다. 그런데 서버 자동 평가, 상태 초기화, 타이머 감소, 세션 종료 리포트 같은 것은 Tool Call이 아닙니다.

그래서 `tool_name`만 있으면 이벤트 분류가 애매해집니다.

## 보완안

```ts
eventType:
  | 'tool_call'
  | 'auto_evaluation'
  | 'state_init'
  | 'state_restore'
  | 'manual_update'
  | 'incident'
  | 'session_end'
```

그리고 `toolName`은 nullable로 두는 게 좋습니다.

```ts
toolName: string | null
```

이렇게 해야 나중에 리포트에서 다음처럼 구분할 수 있습니다.

```txt
- AI Tool Call로 발생한 이벤트
- 서버 자동 평가로 발생한 점수 변화
- 시스템이 자동 복원한 상태
- 세션 종료 시 생성된 리포트 이벤트
```

# 3. `turnIndex` 기준을 더 엄격히 정의해야 함

첨부안에는 `turn_index`가 있지만, 음성 모드에서는 turnComplete, userTranscriptBuffer, AI outputTranscription, response.done 등이 섞입니다.

이걸 명확히 하지 않으면 점수와 이벤트가 어느 사용자 발화에 붙는지 꼬입니다.

## 보완안

```txt
turnIndex는 "사용자 발화 1회 완료"를 기준으로 증가한다.
AI 응답만으로는 turnIndex를 증가시키지 않는다.
텍스트 모드는 user message 저장 시점의 currentTurnIndex를 사용한다.
음성 모드는 userTranscriptBuffer가 확정되어 user.transcription 이벤트가 발생하는 시점을 기준으로 한다.
```

추가로 `turnId`를 두면 더 안전합니다.

```ts
turnId: `${personaRunId}-${turnIndex}`
```

리포트와 이벤트 매칭이 쉬워집니다.

# 4. `timeRemaining` 처리 방식이 빠져 있음

상태 패널에 남은 시간 타이머가 있는데, 실제로 누가 시간을 줄이는지 정의가 없습니다.

방법은 두 가지입니다.

## 방식 A. 클라이언트 표시용 타이머

서버는 `startedAt`, `timeLimitSec`만 내려주고, 클라이언트가 남은 시간을 계산합니다.

장점: 단순함
단점: 서버 상태와 약간 다를 수 있음

## 방식 B. 서버 authoritative timer

서버가 주기적으로 timeRemaining을 갱신합니다.

장점: 정확함
단점: WebSocket/DB 업데이트 부담 증가

## 추천

MVP는 A가 좋습니다.

```ts
simulationState.timer = {
  enabled: true,
  timeLimitSec: 600,
  startedAt: ISODate,
  pausedAt: null,
  elapsedSec: 0
}
```

클라이언트는 이 값으로 남은 시간을 계산합니다.
DB에 매초 저장하지 마세요. 비효율적입니다.

# 5. 텍스트/TTS의 “감정 키워드 감지”는 너무 약함

첨부안에는 텍스트/TTS 모드는 MVP에서 “감정 키워드 감지 등”으로 상태를 갱신한다고 되어 있습니다. 이건 빠르게 만들 수는 있지만 교육용 품질이 낮아질 수 있습니다.

예를 들어 사용자가 “죄송합니다만, 우선 대안을 제시드리겠습니다”라고 말했는데 “죄송” 키워드만 보고 점수를 올리면 부정확합니다.

## 보완안

텍스트/TTS MVP도 최소한 아래 2단계로 가는 게 좋습니다.

```txt
1차: 서버 규칙 기반 fast update
- 너무 짧은 답변: clarity 하락
- 사과/공감 표현 포함: empathy 후보 상승
- 숫자/일정/담당/조치 포함: actionPlan 후보 상승

2차: Gemini Text API 평가
- clarity, empathy, logic, ownership, actionPlan 점수 산출
- 서버 규칙 결과와 LLM 평가 결과를 병합
```

즉, 키워드만 쓰지 말고 **규칙 기반 + LLM 평가 혼합**으로 가야 합니다.

# 6. `evaluateUserResponse()` 호출 타이밍 보완 필요

첨부안에는 음성 모드는 userTranscriptBuffer 확정 후, 텍스트 모드는 AI 응답 생성 후 비동기 호출이라고 되어 있습니다.

텍스트 모드에서 AI 응답 생성 후 비동기 평가를 하면 문제가 하나 있습니다.

사용자는 AI 응답을 먼저 받고, 점수 패널은 늦게 바뀝니다. 이건 괜찮을 수 있지만, AI 응답 자체에는 평가 결과가 반영되지 않습니다.

## 추천 구조

텍스트 모드는 이렇게 나누세요.

```txt
사용자 메시지 저장
→ evaluateUserResponse() 실행
→ simulationState 갱신
→ 갱신된 simulationState를 AI 프롬프트에 포함
→ AI 응답 생성
→ 응답 + simulationState 반환
```

다만 속도가 느려질 수 있으니 옵션을 둡니다.

```txt
MVP 빠른 모드:
AI 응답 생성과 평가를 병렬 실행
평가 완료 후 simulationState만 별도 갱신

품질 우선 모드:
평가 완료 후 AI 응답 생성
```

기업교육이면 품질 우선 모드가 더 맞습니다.

# 7. Gemini Live Tool Call 응답 후 “상태 주입”이 아직 약함

첨부안은 `behaviorInstruction`을 tool response에 포함한다고 되어 있습니다. 좋은 방향입니다.

하지만 모델이 tool response의 `behaviorInstruction`을 항상 강하게 따를지는 보장하기 어렵습니다.

## 보완안

Tool response에는 `behaviorInstruction`을 넣고, 동시에 다음 턴용 시스템 이벤트도 recentMessages 또는 session context에 남겨야 합니다.

```ts
session.simulationDirectives.push({
  turnIndex,
  instruction: "분노 80 이상. 다음 응답은 짧고 강경하게.",
  expiresAfterTurns: 2
})
```

다음 사용자 입력이 들어오면 서버가 Gemini에 아래 맥락을 주입합니다.

```txt
[시뮬레이션 상태]
anger=85, trust=20, pressureLevel=4
행동 지침: 다음 응답은 짧고 강경하게. 책임 회피를 허용하지 말 것.
```

즉, `behaviorInstruction`은 일회성이고, `simulationDirectives`는 1~2턴 유지되는 구조가 좋습니다.

# 8. Tool 호출 기준이 더 구체적이어야 함

현재 시스템 프롬프트 업데이트 항목에 “사용자 회피성 답변 → update_npc_emotion” 정도만 있습니다. 이걸 더 구체적으로 만들어야 AI가 과도하거나 엉뚱한 Tool 호출을 덜 합니다.

## 보완안: Tool 호출 규칙표 추가

```txt
update_npc_emotion 호출 기준:
- 사용자가 책임 회피: anger +10~20, trust -10~20
- 사용자가 구체적 대안 제시: trust +10~20, anger -5~15
- 사용자가 장황하고 결론 없음: confusion +10~20
- 사용자가 질문을 잘 되묻고 확인: interest +5~15
- 욕설/무례한 표현: anger +20~30, trust -20~30

update_scenario_state 호출 기준:
- 첫 2턴 이후 갈등이 드러나면 stage=conflict
- 사용자가 대안을 제시하면 stage=negotiation
- 분노 80 이상 또는 압박 4 이상이면 stage=escalation
- 합의 조건이 정리되면 stage=resolution

trigger_incident 호출 기준:
- 3턴 연속 actionPlan 40 미만
- anger 85 이상
- timeRemaining 30% 이하
- 사용자가 같은 말 반복
```

이 정도는 있어야 시뮬레이션이 예측 가능해집니다.

# 9. `trigger_incident`에 incident taxonomy가 필요함

돌발 이벤트가 자유 텍스트만 있으면 나중에 리포트 분석이 어렵습니다.

## 보완안

```ts
IncidentType =
  | 'executive_join'
  | 'customer_escalation'
  | 'deadline_pressure'
  | 'new_evidence'
  | 'competitor_offer'
  | 'policy_constraint'
  | 'quality_issue'
  | 'manager_interrupt'
  | 'budget_cut'
  | 'compliance_warning'
```

Incident 구조:

```ts
interface Incident {
  id: string;
  type: IncidentType;
  severity: 'low' | 'medium' | 'high';
  message: string;
  turnIndex: number;
  triggeredBy: 'gemini_tool' | 'server_rule' | 'admin';
  createdAt: string;
  resolved?: boolean;
}
```

이렇게 해야 리포트에서 “어떤 유형의 압박 상황에서 약했는지” 분석할 수 있습니다.

# 10. 상태 업데이트 방식은 `statePatch` 중심으로 명시해야 함

첨부안에는 `stateBefore`, `stateAfter`가 있지만, 실제 업데이트는 patch 형태가 더 안전합니다.

## 보완안

```ts
interface SimulationStatePatch {
  npcEmotions?: Partial<NpcEmotions>;
  stage?: ScenarioStage;
  pressureLevel?: number;
  timer?: Partial<TimerState>;
  currentScore?: number;
  incidentsToAdd?: Incident[];
  turnScoresToAdd?: TurnScore[];
}
```

엔진은 항상 이렇게 처리합니다.

```txt
currentState + validatedPatch → nextState
```

장점:

* 상태 전체 덮어쓰기 방지
* race condition 감소
* 로그 비교 쉬움
* WebSocket으로 작은 payload 전송 가능

# 11. 동시성 문제 처리가 빠져 있음

음성 모드에서는 동시에 이런 일이 생길 수 있습니다.

```txt
Gemini Tool Call
사용자 barge-in
turnComplete
평가 함수 완료
DB 저장 완료
WebSocket close
```

이때 상태가 꼬일 수 있습니다.

## 보완안

`simulationState.version`을 넣으세요.

```ts
simulationState.version: number
```

업데이트할 때마다 증가시킵니다.

DB 저장 시:

```txt
현재 version이 내가 읽은 version과 같을 때만 update
다르면 최신 state를 다시 읽고 merge
```

간단 MVP에서는 optimistic locking까지 안 하더라도, 최소한 event log에는 version을 남겨야 합니다.

```ts
stateVersionBefore
stateVersionAfter
```

# 12. 클라이언트 Incident Queue 설계가 조금 더 필요함

`incidentQueue`가 훅에 들어간 건 좋습니다. 다만 같은 incident가 API 응답과 WebSocket에서 중복으로 올 수 있습니다.

## 보완안

incident에는 반드시 `id`를 넣고, 클라이언트에서 중복 제거해야 합니다.

```ts
if (shownIncidentIds.has(incident.id)) return;
```

또한 severity별 색상은 좋지만, 접근성까지 고려하면 색상만으로 구분하지 말고 아이콘/라벨도 넣는 게 좋습니다.

```txt
LOW: 정보
MEDIUM: 주의
HIGH: 긴급
```

# 13. 피드백 리포트에 “결정적 순간” 정의가 필요함

첨부안에는 결정적 순간 하이라이트가 있습니다. 좋은데 기준이 없습니다.

## 보완안

결정적 순간은 아래 조건 중 하나로 잡으면 됩니다.

```txt
- anger가 한 턴에 20 이상 상승
- trust가 한 턴에 20 이상 하락
- turnScore가 이전 턴 대비 25 이상 하락
- trigger_incident 발생 턴
- stage가 escalation으로 전환된 턴
- actionPlan 점수가 30 미만인 턴
- 최종 resolution 직전 턴
```

리포트에 이렇게 보여주면 좋습니다.

```txt
결정적 순간 #2 — 고객 분노 급상승
사용자 발화: "그건 저희 책임은 아닌 것 같습니다."
상태 변화: anger 55 → 82, trust 48 → 25
해석: 책임 회피로 인식되어 압박 단계가 상승했습니다.
개선 문장: "책임 소재를 떠나 고객사 일정 영향부터 우선 줄이겠습니다..."
```

이게 교육적으로 훨씬 강합니다.

# 14. `OpenAI Provider Tool Calling`은 Out of scope지만 fallback 기준은 필요함

첨부안에는 OpenAI Provider Tool Calling 어댑터를 2차 이후로 뺐습니다. 맞습니다.

다만 현재 RoleplayBot은 `aiServiceFactory`에서 OpenAI Provider도 선택될 수 있는 구조입니다. 따라서 텍스트/TTS에서 모델이 OpenAI로 설정되어 있으면 어떻게 할지 명확해야 합니다.

## 보완안

```txt
텍스트/TTS MVP에서는 Provider 종류와 무관하게 Simulation Engine은 서버에서 별도 실행한다.
LLM 평가는 getAIServiceForFeature('conversation') 또는 별도 getAIServiceForFeature('evaluation')를 사용하되,
Provider가 실패하면 규칙 기반 평가로 fallback한다.
```

가능하면 평가용 feature를 분리하세요.

```ts
AIFeature = 'conversation' | 'feedback' | 'strategy' | 'scenario' | 'simulation_evaluation'
```

# 15. 평가 비용/지연 관리가 빠져 있음

매 턴마다 Gemini Text API로 5개 항목 평가를 하면 비용과 지연이 생깁니다. 특히 음성에서는 턴이 많아질 수 있습니다.

## 보완안

```txt
- 10자 미만 발화는 평가 생략 또는 low-confidence 처리
- 동일 발화 반복은 캐시
- 평가 timeout 3초
- 실패 시 rule-based fallback
- evaluationConfidence 필드 저장
```

TurnScore에 넣으면 좋습니다.

```ts
evaluationMethod: 'llm' | 'rule' | 'hybrid'
evaluationConfidence: number
```

# 16. 개인정보/로그 저장 정책도 넣는 게 좋음

음성 transcript와 평가 결과는 교육 데이터입니다. 기업 고객이면 민감할 수 있습니다.

필요 시 messageId 또는 turnIndex로 chatMessages와 연결한다.
관리자 리포트에서는 개인 식별정보 표시 범위를 제한한다.

이건 기업형으로 가려면 중요합니다.

# 17. 테스트 케이스가 빠져 있음

이 정도 기능은 테스트 없으면 쉽게 망가집니다. 태스크에 테스트 항목을 넣는 게 좋습니다.

## 최소 테스트

```txt
1. update_npc_emotion이 0–100 clamp 되는지
2. pressureLevel이 한 번에 ±1만 바뀌는지
3. trigger_incident가 60초 쿨다운을 지키는지
4. 잘못된 Tool args가 들어와도 세션이 죽지 않는지
5. 새로고침 후 simulationState가 복원되는지
6. 음성→텍스트 전환 후 simulationState가 유지되는지
7. turnScores 평균이 가중치대로 계산되는지
8. incident 중복 표시가 방지되는지
9. DB 저장 실패 시 메모리 상태와 대화가 유지되는지
10. 피드백 리포트에 simulation_events가 반영되는지
```

# 18. 첨부안에 바로 추가할 문장

아래 내용을 그대로 추가하면 좋습니다.

```txt
Additional constraints
- simulationState의 canonical owner는 서버 Simulation Engine이다. 클라이언트와 Gemini는 상태를 직접 확정하지 못하고 patch request만 보낸다.
- 모든 상태 변경은 applySimulationPatch() 단일 함수 경로를 통해서만 수행한다.
- simulationState에는 version 필드를 포함하고, 모든 simulation_events에는 stateVersionBefore/stateVersionAfter를 저장한다.
- Tool Call은 전체 state를 덮어쓰지 않고 statePatch만 요청한다.
- personaRunId를 simulationState와 simulation_events의 기본 FK로 사용한다. scenarioRunId는 집계용으로 함께 저장한다.
- evaluateUserResponse는 timeout 3초, 실패 시 rule-based fallback을 사용한다.
- incident에는 고유 id와 type taxonomy를 포함하고, 클라이언트는 id 기준으로 중복 표시를 방지한다.
- generate_feedback_report는 Tool로 노출하지 않고 세션 종료 서버 프로세스에서만 실행한다.
```

# 최종 평가

첨부안은 **기획 방향은 좋고, MVP 범위도 이전보다 현실적으로 줄어들었습니다.** 특히 Tool을 3개로 줄이고, 평가/리포트 생성을 서버 자동 처리로 돌린 건 맞는 판단입니다. 

다만 실개발 관점에서는 아래 5개가 아직 가장 위험합니다.

1. **DB 저장 기준이 conversations인지 personaRuns인지 불명확함**
2. **turnIndex / timeRemaining / eventType 정의가 부족함**
3. **텍스트/TTS 평가 방식이 “키워드 감지” 수준이라 교육 품질이 낮을 수 있음**
4. **Tool response만으로 Gemini 말투가 안정적으로 바뀐다고 보기 어려움**
5. **동시성, 중복 incident, 평가 timeout, fallback 테스트가 빠져 있음**

한 줄로 정리하면:

> 현재 첨부안은 “무엇을 만들지”는 충분히 좋습니다. 이제는 “상태를 누가 소유하고, 어떤 단위로 저장하며, 실패·중복·지연을 어떻게 방어할지”를 더 명확히 해야 합니다.
